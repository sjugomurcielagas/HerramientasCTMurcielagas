Quiero retomar el desarrollo del módulo Análisis de Partidos del
repositorio HerramientasCTMurcielagas.

En esta primera instancia realizá una revisión técnica y funcional
completa SIN modificar archivos, SIN hacer commit y SIN hacer push.

OBJETIVO El registro del partido debe permitir reconstruir el desarrollo
del juego para obtener: - Estadísticas tradicionales. - Evaluación
individual. - Línea temporal. - Narrativa analítica generada por IA.

La narrativa buscada es del tipo: “Argentina dominó los primeros minutos
mediante recuperaciones altas y varias secuencias consecutivas de
ataque.”

PRINCIPIOS

-   Registrar debe ser rápido.
-   Un único registro debe servir para múltiples análisis.
-   Las observaciones tácticas deben alimentar la narrativa.
-   No registrar acciones irrelevantes.

MODOS

1.  Partido rápido (acciones clave).
2.  Análisis de video (todas las opciones).
3.  Partido interno de concentración (categorías simplificadas).

PANEL

Durante la creación del partido permitir elegir: - Panel único. - Panel
doble para registrar ambos equipos simultáneamente.

JUGADORAS

Agregar en Base de Datos el campo: Nombre corto

Usarlo en: - Análisis de Partidos. - Tablero Táctico.

Si no existe, usar automáticamente el nombre habitual.

Al comenzar el partido seleccionar titulares.

Durante el análisis mostrar únicamente quienes están en cancha.

Implementar cambios rápidos: - sale - entra

Guardar el cambio. El minuto debe ser opcional.

MODELO DE REGISTRO

La unidad principal será una SECUENCIA de una misma jugadora.

Ejemplo válido: Recuperación → Regate → Tiro

No registrar acciones intermedias sin valor estadístico.

Ejemplo: Recupera, conduce, frena, intenta un regate, vuelve a intentar,
remata.

Debe registrarse únicamente: Recuperación → Regate → Tiro

Si da un pase o interviene otra jugadora comienza una nueva secuencia.

TIEMPO

Registrar: - minuto opcional.

Si no existe minuto conservar igualmente el orden cronológico.

No implementar segundos ni cronómetro continuo.

OBSERVACIONES

Cada secuencia debe admitir observaciones tácticas propias.

CALIDAD

Mantener una escala única para todas las acciones evaluables.

RESULTADO

Resultado y calidad deben visualizarse simultáneamente.

ATAJADA

Agregar la acción: Atajada de arquera

Revisar la lógica de calidad de la atajada.

TIRO

Agregar una nueva opción de resultado: Bloqueo

RENDIMIENTO

Analizar:

-   carga inicial;
-   tiempo de guardado;
-   caché;
-   backend;
-   fetch repetidos;
-   localStorage;
-   operaciones bloqueantes.

No asumir que el problema está solamente en el frontend.

No modificar Worker, Apps Script ni contratos API.

ENTREGA

Presentar:

A)  Diagnóstico del estado actual.

B)  Brechas respecto a estos requerimientos.

C)  Propuesta de modelo de datos incluyendo ejemplo JSON de una
    secuencia: Recuperación → Regate → Tiro

D)  Plan de implementación dividido en etapas.

E)  Archivos afectados.

F)  Riesgos de compatibilidad.

RESTRICCIONES

-   No modificar archivos.
-   No hacer commit.
-   No hacer push.
-   No tocar Worker.
-   No tocar Apps Script.
-   No cambiar endpoints.
-   No rediseñar completamente el módulo.
-   Mantener compatibilidad con registros existentes.
